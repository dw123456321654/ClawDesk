## 三、风险评估矩阵

### 3.1 评估标准

**影响程度：**
- 高（3）：严重影响项目交付或核心功能
- 中（2）：影响部分功能或进度
- 低（1）：影响较小，可快速解决

**发生概率：**
- 高（3）：发生概率 > 60%
- 中（2）：发生概率 30%-60%
- 低（1）：发生概率 < 30%

**风险等级：**
- 严重（8-9）：必须立即处理
- 高（5-7）：需要重点关注
- 中（3-4）：需要监控
- 低（1-2）：可接受

## 四、风险应对策略

### 4.1 P0 级别风险应对

#### R09 任务拆分粒度过大导致上下文溢出

| 属性 | 内容 |
|------|------|
| 风险等级 | 严重 (9) |
| 影响 | 任务执行失败，代码混乱，需要重做 |
| 应对策略 | **预防措施：** <br>1. 严格执行任务拆分规范，每个任务 ≤ 1 天工作量 <br>2. 产品经理在交付需求前必须拆分为小任务 <br>3. 开发前确认任务粒度是否合理 <br><br>**应急措施：** <br>1. 发现任务过大时立即停止，重新拆分 <br>2. 使用检查点机制保存进度 |

---

### 4.2 P1 级别风险应对

#### R05 Rust 开发经验不足导致后端开发延期

| 属性 | 内容 |
|------|------|
| 风险等级 | 高 (6) |
| 影响 | 后端开发延期，影响整体进度 |
| 应对策略 | **预防措施：** <br>1. 提前学习 Tauri 官方文档和示例 <br>2. 参考 Excel公式助手项目的 Rust 代码 <br>3. 复杂逻辑优先使用 TypeScript，Rust 仅封装命令调用 <br><br>**应急措施：** <br>1. 遇到 Rust 难题时，先跳过，用 TypeScript 实现临时方案 <br>2. 简化后端逻辑，减少 Rust 代码量 |

#### R02 OpenClaw CLI 命令变更导致兼容性问题

| 属性 | 内容 |
|------|------|
| 风险等级 | 高 (6) |
| 影响 | 服务管理功能失效 |
| 应对策略 | **预防措施：** <br>1. 锁定 OpenClaw 版本，记录当前使用的 CLI 命令格式 <br>2. 使用 `openclaw --help` 验证命令可用性 <br>3. 封装命令调用层，便于适配变更 <br><br>**应急措施：** <br>1. 发现命令变更时，立即更新封装层 <br>2. 提供版本兼容性检查 |

#### R07 工作量评估不准确导致延期

| 属性 | 内容 |
|------|------|
| 风险等级 | 高 (6) |
| 影响 | 项目延期交付 |
| 应对策略 | **预防措施：** <br>1. 采用 PERT 估算（乐观/悲观/最可能） <br>2. 预留 20% 缓冲时间 <br>3. 每周评估进度，及时调整计划 <br><br>**应急措施：** <br>1. 发现延期风险时，削减 P2 功能 <br>2. 必要时延长项目周期 |
